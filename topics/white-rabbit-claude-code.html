AWS MCP Server llega a GA con run_script sandboxed y 20+ skills que tu agente no usará sin el rules file

El 6 de mayo AWS anunció la disponibilidad general del AWS MCP Server, un servidor MCP gestionado que da a los coding agents acceso autenticado a cualquier API de AWS con IAM como guardarraíl. La pregunta que recogen del propio equipo del producto: «¿cómo le das a un agente acceso real autenticado a AWS sin entregarle las llaves del reino?». La respuesta GA tiene tres piezas que vale la pena examinar de cerca, más una trampa que te puede dejar el agente sin usar nada de esto.

Instalación y modelo de auth

Para Claude Code, el comando es directo (requiere uv instalado previamente):

claude mcp add-json aws-mcp --scope user   '{"command":"uvx","args":["mcp-proxy-for-aws@latest","https://aws-mcp.us-east-1.api.aws/mcp","--metadata","AWS_REGION=us-west-2"]}'

El truco: el servidor MCP soporta solo OAuth 2.1, pero AWS sigue siendo IAM + SigV4. La pieza mcp-proxy-for-aws corre local y traduce credenciales IAM en requests OAuth-compatibles. Lo importante operativamente: no hace falta un permiso IAM extra para usar el servidor; se pagan solo los recursos que el agente cree. El audit trail completo cae en CloudTrail y las métricas del servidor en CloudWatch bajo namespace AWS-MCP, separadas del uso humano. Esa separación es lo que permite a un equipo de plataforma decir «esto lo ha hecho el agente, no Aina» con evidencia.

Regiones day-one: us-east-1 (N. Virginia) y eu-central-1 (Frankfurt). Solo dos. Para una empresa que tiene producción en ap-southeast o sa-east, eso significa esperar.

run_script: el sandboxed Python que cierra el round-trip

La novedad operativa más interesante respecto al preview es run_script. Permite al agente escribir scripts Python cortos que se ejecutan server-side en un entorno aislado. Características clave:

  • Hereda los permisos IAM del usuario; no añade superficie.
  • Cero acceso de red fuera del propio entorno AWS.
  • Sin acceso al filesystem local ni al shell.
  • Permite encadenar múltiples llamadas a APIs de AWS en un solo round-trip.

El cambio cualitativo: antes el agente describía cinco pasos («lista los buckets, filtra los que tienen versioning, suma sus tamaños...») y cada paso era un tool_use con su latencia y su token cost. Ahora escribe diez líneas de Python que hacen las cinco cosas en una sola llamada. Reduce tokens y reduce ventanas de fallo.

Agent skills: 20+ procedimientos validados (con trampa)

AWS reemplaza el viejo concepto de Agent SOPs (volcados de texto frágiles) por skills: procedimientos curados y mantenidos por los service teams. Más de 20 skills día uno cubren patrones como elegir entre HTTP API y REST API en API Gateway, configurar permissions Lambda correctamente, o decidir entre S3 Vectors y OpenSearch. El agente las carga bajo demanda con retrieve_skill, evitando reventar la ventana de contexto.

Hay un detalle silencioso que cuesta el caso de uso entero: el agente no las usa proactivamente sin un rules file. El repo del Agent Toolkit incluye un fichero de 17 líneas en rules/ que instruye al agente a preferir el MCP server sobre su conocimiento entrenado, a comprobar skills antes de empezar tareas, y a usar retrieve_skill cuando hay duda. Sin ese fichero, un agente discutiendo arquitectura serverless devuelve tres opciones genéricas. Con él, llega a recomendaciones concretas (HTTP API sobre REST API, con la razón arquitectónica). El gap entre tener herramientas y usarlas requiere instrucción explícita.

Lo que conecta con lo que ya sabíamos

En el briefing del 26 de mayo apuntábamos que AWS no era provider day-one de los self-hosted sandboxes que Anthropic anunció en Code with Claude London. Esto no contradice ese punto: el AWS MCP Server es una pieza distinta, no un sandbox para correr agentes. Es la capa de acceso autenticado desde el agente hacia AWS, sin sustituir el harness donde el agente vive. Quien necesite ambas cosas en AWS sigue esperando.

Qué probar esta semana

Si trabajas en AWS y usas Claude Code, son cuatro pasos: instala uv, añade el MCP con el comando de arriba, clona el rules file del Agent Toolkit en .claude/, y comprueba con un prompt como «dame opciones de arquitectura serverless para procesar 10k eventos/segundo». La diferencia entre antes y después de cargar el rules file es la prueba más sencilla de que el setup funciona. CloudWatch debería empezar a mostrar invocaciones bajo AWS-MCP en cuestión de minutos.

Pregunta abierta: AWS dice que la decisión de qué skill cargar la toma el agente. ¿Llegará un equivalente al alwaysLoad de Claude Code para forzar skills críticas (compliance, security) en todas las sesiones, en lugar de fiarlo al juicio del modelo?

MCP 2026-07-28 release candidate: el protocolo se queda sin sesiones y se vuelve plano detrás de un load balancer

El 21 de mayo se cerró el lock del nuevo MCP 2026-07-28 release candidate, la primera revisión mayor de la spec desde noviembre. Hasta el 28 de julio (fecha de publicación final) los SDK maintainers y client implementers tienen diez semanas para validar contra cargas reales. El cambio gordo: MCP es ahora stateless a nivel de protocolo. Lo demás se deriva de esa decisión.

Adiós initialize, adiós Mcp-Session-Id

Hasta la spec 2025-11-25 todo cliente tenía que hacer un handshake initialize y recibía un header Mcp-Session-Id que lo pegaba a una instancia concreta del servidor. Eso obligaba a infra de producción a montar sticky sessions, un store compartido, o deep packet inspection en el gateway para enrutar bien. Un servidor MCP remoto serio era un dolor operacional.

El RC elimina ambos mecanismos. La información del cliente y la versión del protocolo viajan en _meta en cada request. Si el cliente necesita saber capacidades, llama a un método nuevo server/discover. Cualquier instancia del servidor puede responder a cualquier request. Cita textual de la spec: «un servidor MCP remoto que antes necesitaba sticky sessions, store compartido y deep packet inspection en el gateway puede ahora correr detrás de un round-robin plano».

Tres headers que hacen la vida de la infra mucho más fácil

Routable, cacheable, traceable. Tres mejoras que cambian cómo despliegas MCP en una empresa de verdad.

  • Mcp-Method y Mcp-Name headers: load balancers y gateways pueden rutear sin parsear el body. Esto significa que un L7 estándar tipo HAProxy o Envoy hace su trabajo sin reglas custom.
  • Cache control: list y resource reads devuelven ttlMs y cacheScope, modelados sobre HTTP Cache-Control. El cliente entiende ventanas de frescura y límites de compartición seguros sin inventar convenciones.
  • Distributed tracing: propagación de W3C Trace Context vía traceparent, tracestate y baggage. Sigue el hilo que llevamos abierto desde mayo sobre trazado distribuido en Agent SDK; ahora también está en MCP a nivel de spec.

Multi-round-trip sin SSE

Una de las partes feas del MCP viejo: si el servidor necesitaba más input del usuario a mitad de una llamada, mantenía un stream SSE abierto. En un mundo stateless eso ya no encaja. La nueva forma: el servidor devuelve un InputRequiredResult con el prompt; el cliente recoge la respuesta y vuelve a llamar con inputResponses y un requestState echoed. Cualquier instancia puede continuar. Sin streams pegados a una conexión TCP.

Extensions framework: MCP Apps y Tasks salen del core

Las extensiones ganan governance formal: identificadores reverse-DNS, repos separados, versiones independientes y track propio en el proceso SEP. Dos extensiones oficiales arrancan el día uno:

MCP Apps (que veíamos en research preview desde el SEP-1865 en marzo) entra como extensión oficial. Servers entregan HTML interactivo renderizado en iframes sandboxed; las tools declaran UI templates para prefetch y revisión de seguridad. La UI renderizada se comunica con el servidor vía el mismo JSON-RPC, así que los logs de auditoría no tienen huecos.

Tasks gradúa de experimental a extensión. Modelo de ciclo de vida reformulado sobre statelessness: el servidor responde a tools/call con un task handle; el cliente conduce el progreso vía tasks/get, tasks/update y tasks/cancel. Server-directed creation: el servidor decide cuándo una llamada se convierte en task. Detalle importante: tasks/list desaparece porque sin sesiones no hay scoping natural.

OAuth con más rigor

Seis SEPs aprietan la alineación con OAuth 2.0 y OpenID Connect. Lo que va a tocar a quien escriba clientes: validar el parámetro iss por RFC 9207; Dynamic Client Registration ahora exige declarar application_type; las credenciales se vinculan al servidor que las emitió y hay que re-registrar al migrar; flujos OpenID Connect refresh token documentados; .well-known discovery clarificado. No es revolucionario pero es lo que distingue un cliente que pasa una auditoría de uno que no.

Deprecaciones anunciadas

Tres features entran en deprecación con política formal de doce meses mínimo antes de borrar:

  • Roots → parámetros de tool, URIs de resource o config del servidor.
  • Sampling → integración directa con la API del proveedor de LLM.
  • Logging → stderr o OpenTelemetry.

Son cambios annotation-only: los métodos siguen funcionando durante un año mínimo, eliminarlos requiere un SEP separado. La política de doce meses fija expectativas claras para SDK maintainers.

Schema y errores

Tool schemas soportan JSON Schema 2020-12 completo: oneOf, anyOf, allOf, conditionals, referencias. Output schemas son ilimitados; structuredContent acepta cualquier JSON value. Aviso de seguridad: externos no deberían auto-resolverse, hay que limitar profundidad y tiempo de validación. Cambio menor pero relevante: el código de error para recurso no encontrado pasa del custom -32002 al JSON-RPC estándar -32602 Invalid Params.

Qué vigilar las próximas diez semanas

El draft está en modelcontextprotocol.io/specification/draft. La expectativa es que los SDK Tier 1 (Python, TypeScript, Go) soporten la spec dentro del periodo. Para quien tenga un servidor MCP en producción: revisa si dependes de Mcp-Session-Id explícito o de Sampling/Roots/Logging. Para quien construya clientes nuevos: empieza por la spec draft, no por la 2025-11-25; la fecha de salida del RC al final es 28 de julio.

Claude Code 2.1.139: agent view, /goal y los hooks aprenden a no romper el turno

El 11 de mayo Anthropic publicó Claude Code 2.1.139, una de las releases con más cambios estructurales del trimestre. Tres novedades merecen atención inmediata porque cambian cómo se opera con varias sesiones a la vez, cómo se delega trabajo largo, y cómo se escriben hooks que no aborten el turno por error.

Agent view: una pantalla para todas las sesiones

Hasta ahora cada sesión de Claude Code vivía en su propio terminal o worktree. Si tenías cuatro tareas largas en paralelo, necesitabas cuatro ventanas y memoria propia para saber cuál estaba bloqueada esperando tu input. La nueva claude agents (research preview) abre una lista única con todas las sesiones agrupadas por estado: corriendo, esperando aprobación tuya, o terminadas. Reemplaza el típico tmux con cuatro paneles por una vista canónica que el propio binario mantiene.

El cambio importa porque la unidad operativa real de Claude Code en equipos serios ya no es la sesión individual sino el portfolio de sesiones simultáneas. Antes había que construir esa vista a mano con scripts. Ahora viene de fábrica.

/goal: condición de salida en vez de turno único

El comando /goal redefine lo que significa «terminar». En lugar de iterar turno a turno hasta que tú decidas parar, declaras la condición que debe cumplirse y Claude trabaja autónomo hasta lograrla o quedarse atascado. Ejemplo concreto:

/goal All TypeScript errors resolved, tests passing, no ESLint warnings

Funciona en modo interactivo, con -p para scripts, y en Remote Control. Un overlay vivo muestra tiempo transcurrido, número de turns y tokens consumidos acumulados. La regla práctica que sale de la documentación: el goal tiene que ser verificable. «Mejora el código» no es un goal, es una conversación. «Lighthouse performance >90 en /checkout» sí lo es.

Avisos claros: no lo uses para tareas creativas abiertas, monitoriza los primeros goals para estimar coste, y ten un plan B si la condición no se cumple en N turns. El overlay con tokens existe precisamente para que veas la factura crecer en tiempo real antes de que te explote.

Hooks con args y continueOnBlock

Dos mejoras al sistema de hooks que llevaban tiempo pidiendo a gritos.

La primera es el campo args: string[]: hasta ahora los hooks tipo command pasaban por shell, lo que obligaba a comillar rutas con espacios y dejaba la puerta abierta a bugs de inyección. Con args el binario se llama directo, sin shell:

{
  "type": "command",
  "args": ["python3", "/.claude/hooks/validate.py", "--strict"],
  "timeout": 60
}

La guía oficial: usa command cuando necesites pipes, redirección, && o expansiones; usa args cuando llamas a un binario con parámetros. Lo segundo es lo más seguro y debería ser el default en hooks nuevos.

La segunda es continueOnBlock para PostToolUse. Hasta 2.1.138, si un hook devolvía "decision": "block" el turno se abortaba en seco. Con continueOnBlock: true el motivo del bloqueo se devuelve a Claude como un tool_result, y el modelo puede leer la queja y reintentar con otra estrategia:

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/policy-check.py",
        "continueOnBlock": true
      }]
    }]
  }
}

El criterio: activa continueOnBlock cuando el motivo del bloqueo es algo accionable («ese fichero es read-only, escribe en otro»); déjalo desactivado cuando el bloqueo debe parar el turno completo (compliance, seguridad). Hasta ahora muchos hooks de policy actuaban como minas: bloqueaban sin dar contexto y forzaban al usuario a relanzar. Esto convierte el hook en una conversación.

Detalles que también van dentro

MCP stdio servers reciben ahora CLAUDE_PROJECT_DIR en su entorno, llegando a paridad con hooks. Plugin configs aceptan ${CLAUDE_PROJECT_DIR} en command definitions. Transcript navigation suma atajos: ? lista shortcuts, { y } saltan entre prompts del usuario, v alterna el panel. claude plugin details <name> muestra inventario de componentes del plugin y coste de tokens proyectado por sesión, útil antes de instalar un plugin gordo. Las requests de subagent ahora llevan headers y span attributes OTEL propios, lo que cierra el tema de trazado distribuido que vimos en 2.1.121.

Próxima pregunta abierta: ¿agent view sale de research preview en 2.1.155, o se queda como flag durante semanas? El patrón de Anthropic con features grandes ha sido tres a cuatro releases en preview antes de promocionar.

Claude Code 2.1.149: /usage por categoria y el PowerShell cd que saltaba el sandbox sin que te enteraras

La 2.1.149 (22 de mayo) trae dos cosas que vale la pena entender bien: un upgrade al comando /usage que por fin te dice donde se va el presupuesto, y un fix de seguridad en la PowerShell tool que llevaba meses permitiendo escapes silenciosos del workspace.

/usage por categoria: el desglose que faltaba

Hasta esta version, /usage mostraba un total agregado: cuanto contexto consumido, cuanto del limite mensual. Util para saber si te quedabas sin cuota, inutil para diagnosticar por que. Ahora el comando rompe el numero en cuatro categorias:

  • Skills: cuanto contexto y cuantas llamadas consume cada SKILL.md instalado. Si tienes 30 skills cargados y el 80% del coste viene de tres que ni usas, este desglose te lo dice.
  • Subagents: cuanto consumen las invocaciones a sub-agentes (Agent tool calls). Util para detectar workflows donde el orquestador esta delegando demasiado o donde un subagente especifico esta entrando en loops largos.
  • Plugins: agregado por plugin instalado, no por tool individual. Sirve para auditar plugins de terceros antes de mantenerlos en produccion.
  • MCP servers: por server individual, no solo agregado. Esto es lo mas util del set. Un setup tipico de 5 MCP servers con ~58 tools mete ~55,000 tokens al system prompt antes de la primera turn. Saber cual server especifico esta empujando ese numero te permite decidir cuales mover a alwaysLoad: false o directamente desinstalar.

Combinado con el SLASH_COMMAND_TOOL_CHAR_BUDGET y el skillOverrides que llego en 2.1.129, ahora hay herramientas reales para hacer presupuesto de contexto en proyectos grandes. Antes era todo intuicion.

El bug de PowerShell cd que rompia el sandbox

El fix de seguridad mas relevante: en Windows, las funciones built-in de PowerShell cd.., cd\, cd~ y los cambios por letra de drive (C:, D:) cambiaban el working directory sin disparar los checks de permisos del sandbox. El resultado: si un agente tenia permitido leer dentro de C:\proyectos\miapp, podia ejecutar cd.. seguido de un Get-Content y leer fuera del workspace sin que el sistema de permisos se enterara.

Esto es un escape de workspace clasico y llevaba activo desde que la PowerShell tool entro en GA (alrededor de 2.1.84 en marzo). Si llevas usando Claude Code en Windows con permisos estrictos, conviene asumir que cualquier transcript de los ultimos dos meses pudo haber leido archivos fuera de su scope si el modelo decidio explorar. No hay evidencia publica de explotacion en agentes, pero el patron de ataque es trivial: un prompt injection en un README puede pedirle al agente que haga cd ~\secrets y describa lo que ve.

El fix detecta y bloquea estas cuatro formas de redirect. Updateando a 2.1.149 cierra el agujero, pero si tienes hooks PreToolUse propios que filtran comandos de bash, conviene revisarlos: las heuristicas que buscaban cd con espacio probablemente no matcheaban cd.. o C:, asi que la cobertura era doblemente debil.

Otros cambios que se notan

  • /diff scrollable con teclado: flechas, j/k vim-style, PgUp/PgDn, Space, Home/End. Pequeno pero hace usable revisar diffs largos sin tener que extenderlos al output principal.
  • Renderizado de GFM task lists: ahora - [ ] todo y - [x] done aparecen como checkboxes reales en lugar de bullets planos. Si tu agente genera planes con checklists, se vuelven legibles a primera vista.
  • allowAllClaudeAiMcps (enterprise): managed setting para cargar conectores cloud de claude.ai junto al managed-mcp.json local. Hasta ahora era uno u otro.
  • Worktree write allowlist: el sandbox no aplicaba bien los permisos de escritura sobre worktrees adicionales. Si tu proyecto usa worktrees para code review (uno principal mas N branches), conviene reinstalar permisos.

Que vigilar

La 2.1.148 fue un hotfix porque la 2.1.147 devolvia exit code 127 en todos los comandos de bash. Salio y se reemplazo en el mismo dia. Si tu CI ancla a versiones exactas de Claude Code (lo cual recomiendo en pipelines), evita pinear minor versions sin verificar y mantente al menos en 2.1.149.

El otro hilo abierto: con el /usage por MCP server ya granular, falta el equivalente por hook. Si tienes hooks que disparan en cada PostToolUse, no hay manera obvia de saber cuanto suman al coste total. Vale la pena pedirlo en feedback antes de la 2.1.160.

MCP tunnels y sandboxes self-hosted: Anthropic mueve los agentes dentro del perimetro del cliente

El 19 de mayo en Code with Claude London, Anthropic anuncio dos cambios estructurales para Claude Managed Agents que merecen lectura cuidadosa si construyes agentes para empresa: self-hosted sandboxes en beta publica y MCP tunnels en research preview. La idea de fondo es trasladar la ejecucion de tools y el acceso a datos privados al lado del cliente, dejando solo la orquestacion en infraestructura de Anthropic.

Como funcionan los MCP tunnels

Hasta ahora, si querias que un agente alcanzara un MCP server detras de tu firewall, tenias dos opciones malas: exponer el server con un endpoint publico (con OAuth, WAF y todo lo que implica) o renunciar al agente para esos datos. Los tunnels rompen ese dilema con un patron clasico de salida: un gateway ligero que tu despliegas dentro de la red, que abre una conexion saliente unica y encriptada hacia Anthropic. No hay puerto entrante, no hay endpoint publico, no hay cambios en el firewall.

El gateway multiplexa todos los MCP servers internos sobre esa unica conexion. La administracion se hace desde el Claude Console (que servers expone, que agentes pueden alcanzarlos). Funciona tanto con Managed Agents como con la Messages API. Cumple la spec MCP 2025-11-25 (transports stdio y Streamable HTTP) y el modelo de autorizacion sigue OAuth 2.1 + PKCE. Lo unico que Anthropic no ha documentado todavia es la suite de cifrado especifica del transporte; conviene esperar a docs antes de meterlo en compliance reviews.

Self-hosted sandboxes: cuatro proveedores, cuatro perfiles

El sandbox es donde el agente ejecuta bash, escribe archivos y llama APIs. Hasta mayo eso vivia en infraestructura de Anthropic. Ahora hay cuatro providers integrados, mas la opcion BYO:

  • Cloudflare: microVMs con controles de red de salida. Cliente dia uno: Amplitude. Encaja si tu prioridad es controlar exactamente que dominios puede contactar el agente.
  • Daytona: workspaces stateful que se pueden pausar y resumir. Cliente dia uno: Clay (su agente "Sculptor"). Pensado para tareas autonomas de dias o semanas donde el contexto debe sobrevivir reinicios.
  • Modal: containers con acceso a CPU/GPU. Cliente dia uno: DoorDash. Para pipelines computacionalmente pesadas (imagen, datos grandes).
  • Vercel: Firecracker microVMs sobre Amazon Linux 2023, arranque rapido. Cliente dia uno: Rogo (finanzas institucionales). Optimo para developer experience y latencia baja al spin-up.
  • BYO: defines tu propio primitivo de sandbox. Pensado para industrias reguladas con runtimes propios.

La division arquitectonica importa: orquestacion (gestion de contexto, recuperacion de errores, planning) sigue en Anthropic. Ejecucion de tools (bash, file writes, llamadas a APIs internas) vive ahora en tu perimetro. Es el mismo modelo que ya usan plataformas como Replit con sus runners, pero formalizado y con SLAs comerciales.

Lo que aun no esta

AWS no figura como provider day-one, lo cual es notable porque la mayoria de los clientes enterprise vive ahi. La memoria de sesion (las features de memory que Anthropic introdujo en marzo) no esta soportada en modo self-hosted todavia: si te mueves a sandboxes propios, pierdes el feature. Los tunnels siguen en research preview y el acceso se solicita; no hay self-serve aun.

Que significa para quien construye sobre Claude Agent SDK

Para gente trabajando con la SDK directa (no Managed Agents) la noticia importa mas por implicacion que por uso inmediato. Los patrones que Anthropic esta empujando (sandbox externo, conexion saliente unica, separacion orquestacion/ejecucion) son los que vas a querer replicar a mano si tu producto necesita aislamiento serio. El gateway de MCP tunnels, en particular, es un buen blueprint: una sola conexion saliente, multiplexacion de servers internos, gestion centralizada. Anthropic muy probablemente publique el gateway como open source o como spec extendida del MCP en los proximos meses; cuando lo hagan, sera el patron recomendado de facto.

El otro angulo: si llevas tiempo evaluando Managed Agents y la falta de control sobre la ejecucion era el blocker, hoy es buen momento para volver a mirarlo. Especialmente Rogo (finanzas) y DoorDash (operations) son casos de uso que antes habrian forzado build-your-own; ahora funcionan en Managed con sandbox controlado por el cliente. Conviene seguir si AWS aparece como provider antes del Q3 y si la memoria de sesion vuelve.

Claude Code 2.1.129: el bug silencioso que cobraba cache de 1 hora y entregaba 5 minutos

Resumen rapido para quien no abrio el changelog estas dos semanas: entre el 4 y el 6 de mayo, Anthropic publico 64 cambios sobre Claude Code en tres releases (2.1.128, 2.1.129, 2.1.131). El mas costoso en sentido literal es un bug de cache que llevaba semanas activo y nadie habia visto.

El problema: cuando una request seteaba cache_control: { type: "ephemeral", ttl: "1h" }, el sistema cobraba la escritura al tier de 1 hora (mas cara que la de 5 minutos) pero el cache expiraba a los 5 minutos igual. Sin logs, sin warning, sin nada en la response que delatara la degradacion. Si tienes un agente largo con un system prompt grande que reusas cada N turnos, llevas semanas pagando la prima sin recibir el beneficio. La 2.1.129 lo arregla, pero conviene revisar facturas de mayo si usabas cache extendido.

Relacionado, en 2.1.128 detectaron que los progress summaries de los subagentes no usaban prompt caching y disparaban una funcion de summary repetidamente aunque el transcript no hubiera cambiado. El resultado era cache_creation aprox tres veces mayor de lo necesario. Si en abril te parecio que los workflows con muchos subagentes habian subido de coste sin razon aparente, ahora sabes por que.

El otro cambio que rompe configuraciones

El descubrimiento automatico del endpoint /v1/models en gateways, que se introdujo en 2.1.126, se ha convertido en opt-in. Si tu gateway no devolvia bien la lista de modelos, Claude Code se quedaba colgado al iniciar. Ahora hay que setear CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1 explicitamente. Si usas Bedrock, Vertex o algun proxy custom y notaste que el /model picker fallaba, esta es la razon.

Otro cambio sutil pero importante: EnterWorktree ya no crea ramas desde origin/<default-branch> sino desde el HEAD local. Antes, si tenias commits sin pushear y abrias un worktree, esos commits desaparecian del nuevo branch. Si dependes del comportamiento viejo en algun hook propio, conviene revisar. Worktrees creados antes del 4 de mayo pueden estar basados en estados no esperados.

Features nuevas mas pequenas pero utiles

  • --plugin-url <url> (2.1.129): te bajas un .zip de plugin para una sola sesion. Util para probar betas sin instalar nada permanente. Compatible con --plugin-dir que ahora tambien acepta .zip directamente.
  • skillOverrides (2.1.129): tres modos para cada skill instalado: off (oculto al modelo y a la UI), user-invocable-only (no aparece en listings proactivos pero si lo invocas con /name funciona), name-only (colapsa la descripcion para reducir context bloat). El tercer modo es el que mas vale la pena: si tienes 30 skills instalados y solo 5 son relevantes para el proyecto actual, este flag salva miles de tokens en el system prompt.
  • Ctrl+R revertido: vuelve a buscar prompts en todos los proyectos (comportamiento pre-2.1.124). Ctrl+S es el que ahora limita a sesion actual. La inversion de 2.1.124 le molesto a suficiente gente para que la deshicieran.
  • /mcp ahora muestra cuantas tools expone cada server conectado y marca los que devuelven cero (suele ser sintoma de OAuth caducado o config mal).

El fix que mas se va a notar en Mac

Hubo una race condition al despertar el Mac que podia desloguear todas las sesiones de Claude Code abiertas simultaneamente. Cuando el OS se dormia y volvia, multiples sesiones intentaban refrescar el OAuth token al mismo instante; la primera invalidaba al resto. Fix en 2.1.129. Sumado al fix de la 2.1.126 que tambien tocaba el ciclo de sleep, parece que finalmente queda cerrado el capitulo de "sesiones que mueren al volver del coffee break".

Esa numeracion fantasma suele indicar que se revirtio algo grave antes del rollout, pero Anthropic no comento. Si alguien probo 2.1.130 brevemente y noto regresiones, vale la pena reportarlas para entender que pasaba ahi.

Anatomia de un SKILL.md production-ready: por que tu descripcion importa mas que el contenido

El error mas comun cuando empiezas con skills en Claude Code es escribir el SKILL.md como si fuera un README: descripcion bonita arriba, instrucciones detalladas abajo. Y luego te encuentras con que Claude nunca lo invoca solo, o lo invoca cuando no toca, o se le olvida a mitad de sesion. Casi siempre es un problema de como esta escrita la metadata, no del contenido. Aqui van los detalles que la documentacion oficial sepulta a mitad de pagina y que separan un skill que funciona de uno que no.

La descripcion es lo unico que tu skill tiene en el contexto

En una sesion normal, Claude Code carga en contexto solo las descripciones de tus skills disponibles, no su contenido. El cuerpo del SKILL.md solo entra cuando alguien (tu o el modelo) lo invoca. Eso significa que la decision "usar este skill aqui" se toma exclusivamente con tu campo description.

Y hay un limite duro: la descripcion (mas el opcional when_to_use) se trunca a 1,536 caracteres en el listing. Adicionalmente, si tienes muchos skills, el budget global es ~1% del context window (default 8,000 chars, configurable con SLASH_COMMAND_TOOL_CHAR_BUDGET). Skills nuevos pueden ver su descripcion abreviada para caber en ese presupuesto.

Conclusion practica: pon el caso de uso clave en la primera frase. No abras con "Este skill ayuda a..." — abre con el trigger. Compara:

--- malo ---
description: Una herramienta que automatiza el proceso de revision
  de pull requests y aplica nuestras convenciones de estilo
  para garantizar consistencia.

--- bueno ---
description: Revisa un PR contra nuestras convenciones de estilo.
  Use cuando el usuario pide "review", menciona un PR, o pide
  feedback sobre cambios pendientes.

Frontmatter que casi nadie toca pero deberia

  • paths: ["src/**/*.ts", "tests/**/*.spec.ts"] — limita la activacion automatica a archivos que matcheen los globs. Si tu skill solo aplica al backend, restringelo. Reduce falsos positivos sin tener que sobrecargar la description.
  • effort: xhigh — overridea el effort de la sesion mientras el skill esta activo. Util para skills que requieren razonamiento profundo (auditorias de seguridad, refactors grandes) sin que el usuario tenga que cambiar /effort manualmente.
  • model: claude-opus-4-7 — fuerza un modelo concreto durante el turn. inherit mantiene el de la sesion.
  • context: fork + agent: Explore — corre el skill en un subagente forked con system prompt distinto. Crucial: solo funciona si el skill tiene una tarea, no solo guidelines. Si el SKILL.md dice "usa estas convenciones" sin pedir nada, el subagente recibe las convenciones y devuelve vacio.
  • disable-model-invocation: true — solo tu lo invocas con /name. Quita la descripcion del contexto del modelo y bloquea preload en subagentes. Imprescindible para deploy, commits, llamadas a APIs externas con side effects.

Inyeccion de contexto pre-prompt

La sintaxis !`comando` dentro del SKILL.md ejecuta el comando antes de que el contenido llegue a Claude. El output reemplaza el placeholder. Esto es preprocesamiento, no una tool call — Claude solo ve el resultado renderizado:

---
name: pr-summary
description: Resume un PR. Use cuando el usuario quiere ver
  los cambios pendientes o preparar release notes.
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Datos del PR
- Diff: !`gh pr diff`
- Comentarios: !`gh pr view --comments`
- Archivos: !`gh pr diff --name-only`

## Tu tarea
Resume este PR en 3 bullets...

Para multilinea, usa ```! en lugar de la forma inline. Y si necesitas que las rutas funcionen desde cualquier lugar (instalado como plugin, personal o project), usa ${CLAUDE_SKILL_DIR} en los comandos.

Lo que pasa con la auto-compactacion

Aqui hay una trampa silenciosa. Cuando la conversacion se compacta, Claude Code re-adjunta solo la invocacion mas reciente de cada skill tras el resumen, conservando los primeros 5,000 tokens de cada uno. El budget combinado es 25,000 tokens, y se llena empezando por el skill mas reciente. Skills viejos pueden caer enteros.

Si despues de una compactacion notas que un skill "ya no esta funcionando", probablemente sigue ahi pero el modelo esta eligiendo otra ruta. Refuerza la description, o re-invoca el skill explicitamente. Otra opcion mas robusta: forzar el comportamiento con un hooks scoped al skill (campo hooks en el frontmatter) en vez de depender de que Claude lo recuerde.

Otros detalles que ahorran horas

  • Live reload: editar SKILL.md de un skill ya conocido se aplica en la misma sesion. Crear un directorio .claude/skills/ top-level que no existia al arrancar requiere reiniciar.
  • Commands se han fusionado en skills: .claude/commands/deploy.md y .claude/skills/deploy/SKILL.md ambos crean /deploy. Si chocan, gana el skill.
  • Precedencia entre niveles: enterprise > personal > project. Plugins usan namespace plugin-name:skill-name, asi que no chocan.
  • --add-dir es excepcion: los skills en .claude/skills/ de directorios anadidos con --add-dir SI se cargan, aunque otros tipos de configuracion (subagentes, commands, output styles) NO. Util para monorepos.

Si arrastras un set de skills que cargas a mano cada vez con /, vale la pena auditar las descripciones una semana entera vigilando cuales se autoinvocan y cuales no. Los que nunca disparan suelen tener descripciones genericas (mias incluidas). Los que disparan demasiado tienen when_to_use que abarca medio mundo. Ajusta, comprueba, ajusta. Eso es lo que separa una libreria de skills decorativa de una que de verdad cambia como trabajas.

Claude Code en Windows ya no necesita Git Bash: el viaje de la PowerShell tool en tres releases

Si trabajas en Windows con Claude Code, probablemente arrastras alguna cicatriz: paths con barras mezcladas que ningun parser entiende, comandos que pasan por Git Bash y se rompen al tocar el registro, scripts .ps1 que fallan por la politica de ejecucion, o el clasico "command not found" cuando el cmdlet existe pero estaba en pwsh y Claude lo intento ejecutar en bash. La PowerShell tool nacio para arreglar todo eso, y se ha consolidado en tres releases consecutivos. Hoy ya esta lista para ser tu shell primario.

El recorrido

  • v2.1.84 (26 marzo 2026): aterriza la PowerShell tool como opt-in preview. Reemplaza el routing de Git Bash con un spawn directo de pwsh.exe o powershell.exe. Activala desde /config > "Tools & Execution" o editando %USERPROFILE%\.claude\settings.json.
  • v2.1.120 (28 abril 2026): Git for Windows deja de ser requisito. Si Claude Code no detecta Git Bash, usa PowerShell como shell tool por defecto. Para instalaciones limpias en maquinas corporativas donde no quieres meter MSYS, esto es enorme.
  • v2.1.126 (1 mayo 2026): cuando la PowerShell tool esta activa, ahora es el shell primario en lugar del Bash builtin. Es decir, los comandos que antes Claude tendia a routear por Bash por inercia ahora van directos a pwsh. Tambien mejoraron la deteccion de borrados peligrosos en Windows — rm -rf C:\ o Remove-Item C:\Windows -Recurse disparan confirmacion explicita.

Como activarlo

El bloque minimo en settings.json:

{
  "tools": {
    "powershell": {
      "enabled": true,
      "version": "pwsh"
    }
  }
}

Pon "version": "pwsh" para PowerShell 7 (el moderno, multiplataforma) o "version": "powershell" para Windows PowerShell 5.1, que sigue viniendo de fabrica. Si trabajas con cmdlets nuevos como ConvertFrom-Json -AsHashtable, ternarios, null-coalescing o pipeline chain operators (&&, ||), tienes que estar en pwsh — el 5.1 no los soporta.

Hay otro detalle que vale la pena mencionar: /env persiste variables entre PowerShell y Bash dentro de la misma sesion. Si configuras /env AZURE_SUBSCRIPTION_ID=xxx, esa variable esta visible tanto en cmdlets PowerShell como en cualquier comando bash que ejecute Claude despues. Soluciona uno de los dolores cronicos del entorno hibrido en Windows.

Lo que se desbloquea

Con pwsh como shell primario, Claude puede hacer cosas que antes requerian workarounds incomodos:

  • Acceso al registro: Get-ItemProperty HKLM:\SOFTWARE\... directo, sin tener que llamar a reg.exe y parsear texto.
  • Active Directory / Azure AD: Get-AzResource, Get-ADUser, Connect-AzAccount con MFA — flujos que en Git Bash eran un tour de force.
  • Scripts .ps1 nativos: los ejecuta sin invocar pwsh -File manualmente.
  • Pipeline de objetos: en lugar de parsear texto con awk/cut, Claude puede tirar Get-ChildItem | Where-Object Length -gt 1MB | Sort-Object LastWriteTime y trabajar sobre objetos.

Lo que sigue pinchando

Tres trampas que vas a encontrar en cuanto le des cana:

  1. Encoding por defecto: Out-File y Set-Content en Windows PowerShell 5.1 escriben UTF-16 LE con BOM, lo que rompe a otras tools que esperan UTF-8. Si Claude genera un fichero que luego otra tool no puede leer, mira el encoding antes de buscar bugs raros. Pasa -Encoding utf8 explicitamente.
  2. Redirecciones 2>&1 con executables nativos: en 5.1, cuando rediriges stderr de un .exe nativo dentro de PowerShell, cada linea queda envuelta en un NativeCommandError y $? se vuelve $false aunque el exit code sea 0. La tool ya captura stderr aparte — no la rediriger.
  3. El limite de 8191 chars de cmd.exe sigue ahi. No es problema de la tool en si, pero si Claude construye prompts gigantes para invocar otro proceso (otra instancia de Claude Code, por ejemplo, en patrones de orquestacion) puede chocar contra el limite. PowerShell 7 lo sortea con Start-Process y splatting, pero hay que pedirselo explicitamente.

Si vienes de meses peleandote con Git Bash en Windows, este es el momento de dar el salto. La pregunta abierta es si Anthropic acabara haciendo la PowerShell tool default tambien cuando Git Bash esta presente — hoy todavia hay un rollout progresivo en esos casos. En cualquier release 2.1.x posterior a este, mira si aparece algo como "PowerShell tool enabled by default on Windows" para confirmar que se cierra el ciclo.

Cuando usar xhigh en Claude Code: el nuevo nivel de effort de Opus 4.7 y por que es el sweet spot

Opus 4.7 trajo un nivel de effort nuevo, xhigh, encajado entre high y max. Sobre el papel parece un detalle de configuracion. En la practica es la primera vez en bastante tiempo que Claude Code te da un mando claro para decidir cuanto razonamiento quemar en una tarea concreta — y los numeros publicados sugieren que es el sweet spot para casi todo trabajo agentico serio.

Los numeros

Los thinking budgets por nivel:

  • medium: ~2k tokens de pensamiento
  • high: ~5k tokens
  • xhigh: ~10k tokens (nuevo)
  • max: ~20k tokens

En el benchmark interno de Anthropic para coding agentico, Opus 4.7 saca ~71% en xhigh consumiendo ~100k tokens totales por tarea, contra ~74.5% en max consumiendo mas de 200k tokens. Es decir, doblar el presupuesto de pensamiento te compra 3.5 puntos de accuracy. Para SWE-bench Pro — el benchmark agentico mas duro ahora mismo — Opus 4.7 marca 64.3%, por encima de GPT-5.4 (57.7%) y Gemini 3.1 Pro (54.2%).

La diferencia cualitativa entre high y xhigh, segun Anthropic, no es solo cuantitativa: bajo xhigh el modelo activa con mas frecuencia el modo de "adaptive thinking deep", reflexiona sobre resultados intermedios y hace backtracking en paths fallidos de tool calls. High tiende a tirar por "acierta a la primera"; xhigh tira por "explora, falla, recupera". Para refactors grandes o bug localization cross-file la diferencia se nota.

Como controlarlo desde Claude Code

Tres formas:

  • /effort xhigh dentro de una sesion para overridear
  • --effort xhigh al lanzar claude desde la CLI
  • El picker que sale con /model

v2.1.117 subio el default de medium a high para usuarios Pro y Max en Opus 4.6 y Sonnet 4.6, asi que probablemente ya estas en high sin haber tocado nada. Para subir a xhigh tienes que hacerlo explicitamente — y solo aplica a Opus 4.7.

Hay un detalle util que llego en v2.1.120: dentro de tus skills puedes referenciar ${CLAUDE_EFFORT}. Eso permite que un mismo SKILL.md ajuste su comportamiento segun el nivel — por ejemplo, un skill de code review puede pedir checks adicionales solo si ${CLAUDE_EFFORT} es xhigh o max:

---
name: code-review
description: Revisa cambios pendientes
---

Realiza una revision estandar (correctness, tests, naming).

Si CLAUDE_EFFORT es xhigh o max, ademas:
- Analiza implicaciones de seguridad
- Verifica posibles race conditions
- Revisa el blast radius de los cambios

El bug de contexto que valia la pena saber

v2.1.121 arreglo un fallo que afectaba especificamente a Opus 4.7 con la ventana de 1M de contexto: Claude Code estaba calculando el porcentaje de uso contra el techo viejo de 200K, asi que /context reportaba valores 5x inflados y la sesion auto-compactaba prematuramente. Si veias compactaciones agresivas con Opus 4.7 en sesiones largas, antes de buscar tu workaround revisa que estes en 2.1.121+. La factura del bug era doble — perdias contexto util y el modelo no podia usar la capacidad real para la que pagas.

Recomendacion practica

Mi heuristica despues de unos dias trasteando:

  • high para edits puntuales, navegacion de codigo, tareas con un solo objetivo claro. Es el default y vale para el 80% de los prompts.
  • xhigh cuando entras a tareas multi-archivo donde el primer plan probablemente no funciona: refactors, migraciones, debugging de bugs no triviales, exploracion de un codebase nuevo. La diferencia frente a high es notable y el coste extra (~2x thinking tokens) es razonable.
  • max casi nunca. Los 3.5 puntos extra sobre xhigh no compensan duplicar el coste a menos que sea un problema realmente irreductible.

La pregunta que queda abierta es si Anthropic acabara exponiendo el thinking budget como un slider numerico en lugar de niveles discretos. Con la mecanica de adaptive thinking activandose o no segun el nivel, los discretos tienen sentido pedagogico — pero alguien con cargas predecibles podria querer un control mas fino. Mientras tanto, xhigh es el nivel que mas paga la pena explorar esta semana.

You are a senior backend architect...

La idea: cada feature en desarrollo es un track, los tracks pueden ejecutarse en paralelo, y si uno se tuerce puedes hacer revert semantico (no solo git revert, sino revertir tambien el contexto y el estado del agente). La idea es resolver el dolor clasico — cada sesion empieza fresca, no hay continuidad entre dias — con un store de memoria que registra entidades importantes (codigo, decisiones, errores) y las re-inyecta en sesiones futuras via lifecycle hooks. Anthropic no ha enseñado publicamente trabajo en esa direccion, pero es la siguiente abstraccion logica.

Trazado distribuido en Claude Agent SDK: como meter cada subagente y cada Bash en el mismo W3C trace

Si llevas tiempo construyendo agentes con el Claude Agent SDK y ya empezaste a tener pipelines reales, tarde o temprano llega la pregunta: ¿por que esta sesion concreta tardo 47 segundos? ¿Fue el modelo pensando? ¿Un MCP server lento? ¿Un subagente que se atasco esperando IO? La respuesta era manual: añadir prints, correlacionar timestamps, mirar mal. Eso cambio.

El SDK soporta ahora trazado distribuido con OpenTelemetry y propagacion de W3C trace context. La pieza interesante no es que exporte spans — eso lo hacian todos los frameworks de agentes desde hace meses — sino como hereda el contexto en subprocesos. Cuando hay un span activo, los procesos Bash y PowerShell que ejecuta Claude Code reciben automaticamente la variable de entorno TRACEPARENT con el W3C trace context del span de la tool. Cualquier script que lea TRACEPARENT puede colgar sus propios spans bajo el mismo trace, lo cual te da observabilidad end-to-end por scripts y comandos que normalmente serian cajas negras.

Funciona en las dos direcciones. Si lanzas Claude Code en modo no-interactivo con claude -p desde un proceso que ya tiene un span abierto (por ejemplo, un job de CI orquestado), Claude Code lee TRACEPARENT y TRACESTATE de su entorno y crea sus interaction spans como hijos del trace del caller. Es decir: tu pipeline de CI puede iniciar un trace, llamar a varios claude -p en paralelo, y todo aparece en una vista unificada en Honeycomb, Tempo o lo que uses.

Para activarlo necesitas dos flags y un exporter:

export CLAUDE_CODE_ENABLE_TELEMETRY=1
export CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1
export OTEL_TRACES_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_ENDPOINT=https://tu-backend:4317
export OTEL_EXPORTER_OTLP_PROTOCOL=grpc

Lo que sale por el otro lado son spans para las operaciones clave: cada query, cada tool execution, cada hook, cada llamada MCP. El SDK acepta ademas un Tracer proporcionado por el usuario, asi que si tu app embebedora ya tiene su propio setup de OpenTelemetry, lo pasas y los spans se integran sin sobrecarga adicional.

Hay una consideracion practica: la SDK Python alcanzo paridad con TypeScript en este area en releases recientes. Lo que esto te permite hacer en produccion, donde antes era casi imposible:

  • Atribuir latencia entre el modelo y las tools. Cada interaction span tiene hijos para cada tool call, y cada tool call tiene atributos con duracion exacta — ahora tambien expuestos via duration_ms en los inputs de PostToolUse y PostToolUseFailure hooks, asi que puedes hacer alerting sin necesidad de un backend de tracing si solo te interesa el threshold.
  • Correlacionar errores cross-system. Un Bash que llama a una API interna y falla — el trace muestra el span del Bash, el span de la API (porque tu API ya estaba instrumentada), y el span del modelo retomando el control con el error.
  • Medir el coste real de subagentes paralelos. Cuando lanzas 5 subagentes en paralelo, ¿cual es el cuello de botella? El trace lo enseña sin discusion.

Hay una segunda capa que conviene saber. Claude Code emite eventos estructurados para casos de uso de logging: claude_code.user_prompt, claude_code.api_request, claude_code.tool_result, claude_code.api_error y claude_code.tool_decision. Esto no es trazado, es logging estructurado, y va por OTEL_LOGS_EXPORTER. La utilidad principal es trackear accept/reject rates de las decisiones del modelo y construir dashboards de uso por usuario o por proyecto. Anthropic captura tambien metricas: tokens in/out, sesiones, coste, lineas de codigo modificadas, PRs creadas, commits y tiempo activo.

Trade-offs honestos. El overhead del exporter es pequeño pero no nulo — flush cycles de 10 segundos para metricas y 5 para logs son los defaults razonables. OTEL_LOG_USER_PROMPTS=1 captura el texto crudo de los prompts en los logs, lo cual es util para debug pero un riesgo de privacidad evidente si trabajas con datos sensibles. Y la beta de enhanced telemetry sigue marcada como beta, asi que la API de spans podria cambiar antes de GA.

¿Que vigilar despues? La integracion con Langfuse, SigNoz y otros backends ya tiene guias propias, pero falta un patron canonico para correlacionar trazas con coste API real — hoy puedes verlo en metricas separadas, no como atributos del span de la query. Claude Managed Agents en beta publica: el harness gestionado para correr Claude como agente autonomo

El 8 de abril Anthropic publico en beta Claude Managed Agents, un harness completamente gestionado para correr Claude como agente autonomo dentro de contenedores en su infraestructura. La idea: en vez de que tu construyas el agent loop, el sandbox y la capa de ejecucion de tools, defines el agente y mandas sesiones por API. Claude orquesta el bucle, ejecuta bash y operaciones de archivo en el container, y te devuelve eventos por SSE.

Donde encaja esto en el ecosistema. Hasta ahora habia dos formas de construir agentes con Claude: (1) la Messages API directa, donde tu escribes el loop completo y manejas tool execution, sandboxing, compaction y cache; (2) Claude Code, que es un harness optimizado para coding pero atado al CLI y al editor. Managed Agents es una tercera via, pensada para tareas largas (minutos u horas), asincronas, que queres correr como servicio sin montar Kubernetes. La diferencia clave con Claude Code: Managed Agents corre en infra de Anthropic y se factura como API, no como suscripcion.

Cuatro conceptos. El modelo mental cabe en una tabla:

  • Agent: la definicion (modelo, system prompt, tools, MCP servers, skills). Se crea una vez y se referencia por ID.
  • Environment: un template de container con paquetes preinstalados, reglas de red y archivos montados.
  • Session: una instancia corriendo del agente dentro de un environment, ejecutando una tarea concreta.
  • Events: los mensajes intercambiados (user.message, agent.message, agent.tool_use, session.status_idle).

El flujo en codigo. Tres llamadas y un stream:

from anthropic import Anthropic
client = Anthropic()

agent = client.beta.agents.create(
    name="Coding Assistant",
    model="claude-opus-4-7",
    system="You are a helpful coding assistant.",
    tools=[{"type": "agent_toolset_20260401"}],
)

environment = client.beta.environments.create(
    name="quickstart-env",
    config={"type": "cloud", "networking": {"type": "unrestricted"}},
)

session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
    title="Quickstart session",
)

with client.beta.sessions.events.stream(session.id) as stream:
    client.beta.sessions.events.send(session.id, events=[{
        "type": "user.message",
        "content": [{"type": "text", "text": "Genera los primeros 20 Fibonacci en fibonacci.txt"}],
    }])
    for event in stream:
        match event.type:
            case "agent.message": print(event.content[0].text, end="")
            case "agent.tool_use": print(f"\n[Using tool: {event.name}]")
            case "session.status_idle": break

El tool type agent_toolset_20260401 habilita el set completo: bash, file ops (read/write/edit/glob/grep), web search, web fetch, MCP servers. Se puede restringir por tool individualmente. Todas las llamadas requieren el header beta managed-agents-2026-04-01 (los SDK lo ponen automatico).

El detalle interesante: steering en vivo. Una sesion no es fire-and-forget. Mientras el agente trabaja podes mandar eventos user.message adicionales que se inyectan en el contexto, o un evento de interrupcion para cambiar la direccion mid-execution. Esto resuelve el problema clasico de que el agente se vaya por las ramas sin que puedas detenerlo: ahora podes corregir el rumbo sin matar la sesion ni perder el estado del filesystem. La transcripcion completa queda persistida server-side y se puede pedir entera por API.

Lo que NO es esto. No es una alternativa a Claude Code para coding interactivo: el branding guidelines de Anthropic prohibe explicitamente llamarlo "Claude Code" o "Claude Code Agent" si lo integras en tu producto. La intencion es clara: Managed Agents es la primitiva para construir productos con tu propia marca encima, no un IDE. Tampoco es un reemplazo de la Messages API: si necesitas control fino sobre el loop, custom tool execution, o latencia ultra-baja para chat sincrono, sigue siendo Messages API.

Pricing y rate limits. Se factura por tokens segun el modelo elegido (claude-opus-4-7, etc.) mas el tiempo de container. Rate limits per organization: 300 req/min para endpoints de creacion (agents, sessions, environments), 600 req/min para reads (retrieve, list, stream). Los limites de spend de cuenta tambien aplican. Memory para Managed Agents tambien entro en beta publica el mismo dia: persistencia entre sesiones del mismo agente.

Casos donde tiene sentido. Pipelines de data engineering que corren horas, agentes de research autonomos que iteran sobre fuentes, jobs nocturnos de auditoria de codigo, asistentes de soporte que abren tickets internos. Cualquier cosa que hoy resuelves con una EC2 corriendo un script con Anthropic SDK + tu propio sandbox + un sidecar de logs es candidata a migrar. Lo que vigilar las proximas semanas: outcomes y multi-agent estan en research preview (hay que solicitar acceso), y el header beta sugiere que la API puede cambiar entre releases. No lo metas en producto critico hasta que salga de beta.

MCP Apps (SEP-1865): tus servers MCP ahora pueden devolver UI interactiva, no solo texto

Hasta ahora un MCP server era una caja que escupia texto: invocas visualize_data y recibes una tabla en markdown que el modelo describe. La extension oficial MCP Apps, formalizada como SEP-1865 y ya en produccion, cambia eso: el server puede devolver un dashboard React, un formulario, una grafica interactiva, y el host (Claude, Claude Desktop, ChatGPT, VS Code Insiders, Goose) lo renderiza dentro de un iframe sandboxed en la conversacion. Y la UI puede llamar de vuelta a otras tools del server.

El mecanismo es deliberadamente simple. Una tool declara que tiene UI asociada via metadata:

{
  name: "visualize_data",
  description: "Visualize data as an interactive chart",
  inputSchema: { /* ... */ },
  _meta: {
    ui: {
      resourceUri: "ui://charts/interactive"
    }
  }
}

Cuando el modelo invoca la tool, el host detecta el campo _meta.ui.resourceUri, hace fetch del recurso bajo el esquema ui:// (que sirve HTML/JS bundleado desde el propio server) y lo monta en un iframe restringido. La UI no es estatica: usa el SDK @modelcontextprotocol/ext-apps para hablar con el host via JSON-RPC sobre postMessage:

import { App } from "@modelcontextprotocol/ext-apps";

const app = new App();
await app.connect();

// La UI puede llamar tools del server a traves del host
const response = await app.callServerTool({
  name: "fetch_details",
  arguments: { id: "123" },
});

// Y puede inyectar contexto de vuelta al modelo
await app.updateModelContext({
  content: [{ type: "text", text: "User selected option B" }],
});

El segundo metodo es el interesante para sistemas de agentes: updateModelContext permite que la interaccion del usuario con la UI se convierta en contexto que el modelo ve en el siguiente turn. El usuario aprieta un boton, marca opciones, ajusta sliders; eso entra al contexto como mensaje y el agente sigue trabajando con esa informacion. Es la diferencia entre renderizar y conversar.

Por que importa para quienes construyen agentes. Hasta ahora habia dos opciones malas para meter UI en un flujo de agente: (1) renderizar markdown limitado dentro del chat, (2) abrir una webapp aparte y romper el flujo. MCP Apps abre una tercera via: el server expone tanto la logica (tools) como la presentacion (UI) bajo el mismo protocolo, y el host se encarga del sandbox y de la comunicacion. Para casos como wizards de configuracion, dashboards de monitoring, formularios de aprobacion en pipelines, exploracion de datos paso a paso, esto cierra una brecha real.

Modelo de seguridad. Todo el HTML corre en iframes con permisos restringidos (sin acceso al DOM padre, sin acceso a APIs sensibles del navegador). Los templates UI deben pre-declararse para que el host los pueda revisar antes de renderizar. Cada llamada JSON-RPC desde la UI al host es auditable. Y los hosts pueden requerir aprobacion explicita del usuario para tool calls iniciados desde la UI, no solo desde el modelo. Es razonable como diseno por defecto, pero si vas a publicar un MCP App al publico conviene leer el threat model completo antes: el iframe no es Chrome sandbox, hay vectores conocidos en cualquier UI que mezcla privilegios.

Quien lo soporta hoy. Claude (web y desktop), VS Code Insiders, Goose, Postman, MCPJam, y ChatGPT segun el anuncio de noviembre. Es la primera extension oficial del protocolo MCP y se desarrollo conjuntamente entre los maintainers core de Anthropic y OpenAI con el grupo MCP-UI. Esto es importante: las dos compañias mas grandes del ecosistema acordaron un standard antes de fragmentar (OpenAI tenia su propio Apps SDK; ahora confluyen).

Si ya mantienes un MCP server con FastMCP 3.0 (release de enero), añadir una tool con UI es ~50 lineas: una tool con metadata _meta.ui, un endpoint que sirva el bundle bajo ui://, y la UI minima conectada al SDK. Casos donde el ROI es alto: tools que hoy generan tablas largas que el modelo describe mal, formularios de input estructurado donde escribir JSON a mano es infierno, y aprobaciones humanas en pipelines de agentes (mostrar diff, esperar click). El ecosistema todavia es joven (tres meses desde la formalizacion), asi que es buen momento para publicar algo y aprender.

Forked subagents en Claude Code 2.1.117: heredar contexto sin pagar el prompt dos veces

Si has construido algo con el Claude Agent SDK, conoces el dilema de los subagentes: aíslan contexto (genial), pero arrancan con la conversación en blanco. Cada vez que lanzas tres en paralelo para revisar un PR desde ángulos distintos, los tres pagan el mismo prompt de sistema, las mismas instrucciones del CLAUDE.md, los mismos archivos previos. A 50k tokens compartidos, son 150k tokens facturados a precio completo solo para llegar al primer turno útil.

La versión v2.1.117 de Claude Code introdujo una alternativa: los forked subagents. La feature ya estaba en el código pero dead-code-eliminated en releases públicos hasta que se activó este abril. Se habilita con una sola variable de entorno:

export CLAUDE_CODE_FORK_SUBAGENT=1

Con eso activo, cuando llamas al Agent tool sin especificar subagent_type, el hijo no nace fresco: hereda byte por byte el system prompt del padre (vía override.systemPrompt desde toolUseContext.renderedSystemPrompt), su array de tools (con useExactTools: true, saltándose resolveAgentTools()), y todo el historial de mensajes hasta el momento del fork. Los tool-use blocks del padre se reemplazan por un placeholder constante (FORK_PLACEHOLDER_RESULT) para no contaminar la traza.

La economía: cache sharing

Lo interesante no es solo la herencia, sino que el fork comparte el prompt cache con la sesión madre. Como esos tokens ya están cacheados, el primer turno del hijo se factura cerca de la tarifa cache-read, alrededor del 10% del precio de input normal en Sonnet. Los números del análisis de Build This Now con 48.500 tokens de prefijo compartido:

  • Sin fork, cinco hijos en paralelo: 243.500 tokens facturados (cada uno reconstruye el prefijo)
  • Con fork, cinco hijos en paralelo: 68.900 tokens facturados (solo el primer hijo paga retail; los siguientes leen del cache)

Reducción aproximada de 10x por hijo adicional. No es marketing, es aritmética del cache de Anthropic aplicada al patrón de fan-out que cualquiera con tres subagentes en paralelo ya estaba usando.

Cuándo tiene sentido (y cuándo no)

El fork rompe el aislamiento de input que era el argumento original de los subagentes. Eso lo hace ideal para:

  • Explorar varias soluciones al mismo problema en paralelo desde el mismo punto de partida
  • Tareas auxiliares que necesitarían demasiado briefing si arrancaran limpias (revisa este PR teniendo en cuenta los cinco archivos que ya analizamos)
  • Workflows donde el padre ya cargó contexto caro (CLAUDE.md grandes, MCP servers con mucho output)

Pero conviene vigilar varias cosas. Los forks no pueden anidar: hay un guard de dos capas que detecta si querySource === 'agent:builtin:fork' o si el historial contiene el tag XML que declara que el agente ES el fork y no debe spawnear sub-agents. Tampoco son compatibles con --print ni con coordinator mode (un coordinador forkeado intenta orquestar en vez de ejecutar). Y el contexto del hijo escala con la longitud de la sesión madre: si tu sesión principal va por 180k tokens, cada fork arranca con 180k tokens en memoria, aunque la mayoría sean cache reads baratos.

Encaja con el resume de subagentes

Esto se complementa con otra mejora reciente del SDK que muchos pasaron por alto: los subagentes ahora se pueden reanudar. La docs oficial muestra el patrón: capturas el session_id del ResultMessage, parseas el agentId del contenido (aparece como agentId: <uuid> en el resultado del Agent tool) y en una segunda query() pasas resume: sessionId con un prompt tipo Resume agent X. El subagente recupera su transcripción completa, no la memoria del padre, su propia historia.

Combinado, el patrón emergente para sistemas multi-agente serios en el SDK es: fork para fan-out barato cuando los hijos necesitan el contexto del padre, subagentes nombrados con tools restringidos para tareas con límites estrictos de seguridad, y resume para workflows de larga duración donde un revisor especializado vuelve días después.

Qué probar y qué vigilar

Lo más útil de cara a las próximas semanas: medir cuánto baja realmente la factura de un workflow que ya use Agent SDK con varios hijos en paralelo. La estimación teórica de 10x asume que el prompt cache hace su trabajo, y el cache de Anthropic tiene ventana de 5 minutos por defecto. Si tus subagentes tardan más en arrancar (esperando aprobaciones, locks, etc.), el ahorro se evapora. La feature sigue marcada como experimental en el código, así que v2.1.118 en adelante puede cambiar comportamiento. Vale la pena leer las release notes cada semana hasta que salga del flag.
